1: Explain the purpose of each major component in your application 
            architecture (e.g., app.py, HTML/CSS/JS files, MySQL database) 
            and how they interact during a user registration request.

The system follows a decoupled Client-Server architecture:
 Frontend Client (HTML/CSS/JS): 
   - `register.html` provides the interactive form fields for user data collection.
   - `style.css` ensures a clean visual layout and structured formatting.
   - `script.js` intercepts the form submission event, executes client-side validation, converts data into a clean JSON payload, and initiates a network request via the Fetch API to the backend.
 Backend Server (`app.py`): 
   - Uses Flask to expose programmatic API endpoints (`/register`, `/login`, etc.). 
   - Validates input completeness, manages the session lifecycle, hashes plain-text passwords securely using Bcrypt, and handles responses with appropriate HTTP status codes.
 Database Layer (MySQL): 
   - The persistent storage engine running the `auth_system` database schema. 
   - The `users` table enforces transactional data integrity through unique constraints on usernames and emails.


Interaction Flow during a Registration Request:
1. The user inputs their desired credentials into `register.html` and clicks "Sign Up".
2. `script.js` listens to the submit event, packs fields into JSON, and targets the `/register` endpoint via an HTTP POST request.
3. Flask intercepts the incoming request at the `/register` routing layer, checks for missing data, and verifies whether the username or email already exists in the system.
4. If unique, the backend uses Bcrypt to derive a robust password hash string.
5. An SQL insertion statement is constructed and executed inside the MySQL environment using `mysql-connector-python`.
6. Upon a successful transaction commit, Flask sends a JSON response containing an HTTP 201 Created status back to the client.
7. `script.js` reads the success response status and shifts the browser view layer immediately to `login.html`.

2  Detail the security mechanisms implemented in your code to 
            protect sensitive user data (e.g., passwords) both during 
            transit and at rest.

The architecture ensures full-spectrum security enforcement:
 Security at Rest (Data Storage):
   - Cleartext strings are never allowed to enter the database engine.
   - The application leverages a robust password hashing algorithm (Bcrypt via `flask_bcrypt`). 
   - Bcrypt automatically processes input plain text with dynamic salting configurations. This design neutralizes rainbow table attacks or pre-computed hash dictionary matching. Even if an attacker gains illicit access to the raw table data inside MySQL, the cryptographic structure prevents the reversal of hashes back to plain-text assets.
 Security in Transit (Data Transmission):
   - The client packages payload objects strictly within programmatic request bodies using POST operations over the Fetch API instead of leaving credentials visible inside URL query string spaces.
   - Production systems overlay this communication pipeline with standard Transport Layer Security (HTTPS) configurations, completely blocking attempts at man-in-the-middle packet eavesdropping.
 Session Security Management:
   - Authenticated state sessions rely heavily on secure Flask cookie generation layers backed by cryptographic server secret keys, rendering credential spoofing highly unviable.

3  Discuss how your application handles errors and exceptions, 
            such as database connection failures or duplicate username 
            registrations.

 Database Connectivity Isolation:
   - Data infrastructure connections are enclosed inside explicit `try-except-finally` code blocks. 
   - When connection configurations fail (e.g., due to an incorrect root password or offline server states), the system intercepts the error immediately, generates informative internal logs to the terminal, and issues an HTTP 500 Server Error fallback indicator rather than exposing raw system stack traces to end users.
 Duplicate Input Validation:
   - The backend checks database records for existing entries before performing insertions.
   - If a collision occurs against unique indices (username/email), the MySQL execution layer triggers a specific error instance. The application translates this event cleanly into a structural user response containing an HTTP 400 Bad Request status code accompanied by a clear descriptive validation string ("Username or Email already registered").
 Frontend Error Handling:
   - The Javascript code wraps network calls inside robust `try-catch` structures. 
   - If the network fails or the server returns an error, the frontend catches the failure and dynamically updates the DOM layer to display a friendly error message banner directly onto the active page interface.
